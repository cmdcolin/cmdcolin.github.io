The "Ralph loop" is a popular workflow but nobody explains it plainly. The repo
at https://github.com/snarktank/ralph says:

"Ralph is an autonomous AI agent loop that runs repeatedly until all PRD items
are complete."

It goes on to reference the term PRD over and over without ever defining the
acronym, much less explaining what makes a good one.

![](/media/prd2.png)

Despite the cute Ralph Wiggum branding, the authors are overcomplicating
something that's actually really simple. Let me break it down.

## A PRD ("project requirements document") is just a todo list

You don't need anything fancy. Again, a PRD is just a TODO list. A markdown file
with checkboxes works fine:

```markdown
- [ ] Add priority column to tasks table
- [ ] Show priority badge on task cards
- [ ] Add priority selector to task edit modal
- [ ] Filter tasks by priority, persist in URL
```

The agent reads this at the start of each session, picks the top unchecked item,
implements it, and checks it off. If you already have a scrappy handwritten todo
list, just paste it and ask Claude to clean it up. It'll add some structure and
priorities. The "format" really doesn't matter, it's plaintext, and Claude works
well with plaintext.

## How does PRD connect to the "Ralph loop"

Just to mention up front, I don't actually use this...yet.

But it is worth understanding what the idea is. Ralph is a bash loop where each
iteration spawns a fresh Claude instance, hands it the todo list, and tells it
to do one task and stop. The agent commits its work, marks the task done, and
exits, and then the loop spawns a new instance for the next task.

```bash
for i in $(seq 1 $MAX_ITERATIONS); do
  OUTPUT=$(claude --dangerously-skip-permissions --print < CLAUDE.md)
  if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
    exit 0
  fi
done
```

The agent has no idea there's a loop going on. It stops after one task because
the prompt tells it to:

```
Work on ONE story per iteration. If stories remain, end normally.
If all are complete, reply with: <promise>COMPLETE</promise>
```

After each run, the bash script literally greps the output for the string
`<promise>COMPLETE</promise>`, and if it finds it, the loop exits. Otherwise it
just spawns another instance and keeps going. The entire stop condition is a
grep, which is kind of beautiful in its simplicity.

Each fresh instance starts with zero context, and memory between iterations
comes from git history and the updated todo list. Claude Code also has
[auto-memory](https://code.claude.com/docs/en/memory) which accumulates codebase
knowledge across sessions automatically, so the hand-rolled `progress.txt` that
ralph ships with isn't really necessary anymore.

## You be the Ralph

You can play the role of the loop yourself. Start each session with something
like: "read `todo.md`, pick the top incomplete task, implement it, mark it done,
then stop." Then review the result, start a new session, and repeat until you've
worked through the list.

If you want more control, write the task list yourself and ask Claude for a plan
before any code gets written. You can also just type in the Claude interface to
steer it mid-task if something looks off, and redirecting mid-session is totally
fine since you still get most of the benefit of having the todo list driving
things forward.

I'd recommend starting manually and automating once you trust the output.

## Conclusion

The ralph repo adds shell scripts, "skills", a JSON format for its PRD, a
progress log, and a bunch of other stuff. Maybe that was once helpful, but as
we've seen over and over during this AI upheaval, things that seem necessary one
month aren't needed the next.

My current workflow doesn't use any skills or plugins, and I don't even run a
ralph loop, but I think there are good lessons to take from it. Mainly: keep a
todo list in your repo and let the agent work through it.
