The "Ralph loop" is a popular workflow but nobody explains it plainly. The repo
says: "Ralph is an autonomous AI agent loop that runs repeatedly until all PRD
items are complete."

![](/media/prd2.png)

I've been coding for years and had never heard the term PRD. The authors don't
define it either, which makes it sound more complicated than it is. But it is
not magic and actually is best thought of without magic.

It is just a mechanism that you can use to help maximize agentic workflows. A
PRD — "project requirements document" — is basically just a todo list. The Ralph
github repo ships a JSON schema for it, but you don't need any of that.

## The todo list

A markdown file with checkboxes works fine:

```markdown
- [ ] Add priority column to tasks table
- [ ] Show priority badge on task cards
- [ ] Add priority selector to task edit modal
- [ ] Filter tasks by priority, persist in URL
```

The agent reads this at the start of each session, picks the top unchecked item,
implements it, and checks it off. If you have a scrappy handwritten todo list,
just paste it and ask Claude to clean it up — it's nothing magic, it'll add some
structure and priorities. Either version works. The format is not the point.

The maxed-out version: give Claude a vague goal and let it write and manage the
list itself — decomposing, reordering, adding tasks it discovers
mid-implementation. That's giving up a lot of control, but it's the intended end
state of this workflow.

## How the loop works

Ralph is a bash loop. Each iteration spawns a fresh Claude instance, hands it
the todo list, and tells it to do one task and stop. The agent commits its work,
marks the task done, and exits. The loop spawns a new instance for the next
task.

```bash
for i in $(seq 1 $MAX_ITERATIONS); do
  OUTPUT=$(claude --dangerously-skip-permissions --print < CLAUDE.md)
  if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
    exit 0
  fi
done
```

The agent has no idea there's a loop. It stops after one task because the prompt
says so:

```
Work on ONE story per iteration. If stories remain, end normally.
If all are complete, reply with: <promise>COMPLETE</promise>
```

The bash script greps for that tag. No tag = spawn another instance. The agent
never needs to know about the loop for this to work.

Each instance starts with zero context. Memory between iterations comes from git
history and the updated task list. Claude Code's
[auto-memory](https://code.claude.com/docs/en/memory) (v2.1.59+) also
accumulates codebase knowledge across sessions automatically, replacing the
hand-rolled `progress.txt` that ralph ships with.

## Doing this manually

You are the loop. Start each session with: "read `todo.md`, pick the top
incomplete task, implement it, mark it done, then stop." Review the diff, start
a new session, repeat.

**More hands on the wheel:**

- **Write the task list yourself.** Don't let Claude decompose the goal — write
  and order tasks before any code is written.
- **Ask for a plan first.** Have Claude describe its approach before writing
  code, then approve it.
- **Review the diff before continuing.** `git diff HEAD~1` after each session.
  Fix anything wrong before it becomes the base for the next task.
- **Split tasks smaller.** The smaller the task the easier the diff to review.
- **Keep CI green.** Broken state compounds across sessions.

Start manual. Automate once you trust the output.

## Why bother

Even if you never run the ralph loop, just keeping a todo file and having Claude
tick items off it as you go will noticeably speed up how fast you move through a
big feature. Instead of context-switching between "what did I finish, what's
next, did I forget anything," the list handles that. You stay in the flow of
reviewing and approving rather than project-managing.

The full ralph loop makes sense when you want to step further back — kick it off
and come back later to review a batch of completed work rather than babysitting
each task. But most of the benefit comes just from keeping the list in the first
place.
