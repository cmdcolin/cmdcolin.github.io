---
title:
  The thing that people never seem to talk about when they talk about react
  routers
date: 2026-03-19
---

When people reach for React Router, what are they actually trying to do? Usually
they want clean URLs — `/about`, `/settings`, `/user/123`. To make that work in
a deployed SPA, you need a server rewrite rule that sends every URL to
`index.html`, React boots, reads `window.location`, and renders the right thing.
You've told the server a lie: "yes, `/about` is a real page" — when really there
is only one page.

What's remarkable is that this dependency is almost never mentioned. You can
read the React Router
[getting-started guide](https://reactrouter.com/start/framework/routing) end to
end and come away thinking routing is purely a frontend concern. Then you
deploy, type `/about` into the address bar directly or hit refresh, and get
a 404. The server has no idea what `/about` is.

You never noticed this during development because your dev server hid it from
you. Both webpack-dev-server (used by Create React App) and Vite silently
redirect all 404s back to `index.html` via a
[`historyApiFallback`](https://webpack.js.org/configuration/dev-server/#devserverhistoryapifallback)
option that's on by default. The problem is invisible until production.

So you go searching, eventually find a
[fix guide](https://ui.dev/react-router-cannot-get-url-refresh) or a note buried
in React Router's [SPA deployment how-to](https://reactrouter.com/how-to/spa),
and learn that you need to configure your server to rewrite everything to
`index.html`. For Apache that's a `RewriteRule`, for nginx it's a `try_files`
directive, for Netlify it's a `_redirects` file. Every platform has its own way
of expressing the same underlying lie.

So what does the landscape actually look like? There are roughly three ways
people deploy React Router in the real world.

The first is the catch-all rewrite. You build a static bundle, deploy it to a
CDN or a basic web server, and configure a fallback rule so that any URL that
doesn't match a file on disk gets served `index.html` instead. In nginx:

```nginx
location / {
  try_files $uri /index.html;
}
```

In Apache:

```apache
RewriteCond %{REQUEST_FILENAME} !-f
RewriteRule ^ /index.html [L]
```

In a Rails or Django app serving a React frontend, you'd add a catch-all route
at the bottom:

```ruby
# Rails
get '*path', to: 'application#index'
```

```python
# Django
re_path(r'^.*$', TemplateView.as_view(template_name='index.html'))
```

This is the pattern that underpins most small-to-medium React Router
deployments. It is the lie in its purest form.

The second is server-side rendering. Instead of one `index.html` for every URL,
you run a Node.js server that renders the right HTML for each route and sends it
down the wire. React Router ships a `StaticRouter` for exactly this:

```js
// Express
app.get('*', (req, res) => {
  const html = renderToString(
    <StaticRouter location={req.url}>
      <App />
    </StaticRouter>,
  )
  res.send(`<html><body><div id="root">${html}</div></body></html>`)
})
```

This eliminates the catch-all problem: every URL is now a real server endpoint
that returns meaningful HTML. The client hydrates and React Router takes over
from there. This is roughly what Airbnb built, and what frameworks like Next.js
and Remix (now merged into React Router v7) give you out of the box. Shopify,
for instance, uses React Router v7 in this mode for their app platform.

The third is that large companies skip React Router entirely and build their own
— and the reasons why are instructive.

Meta
[rebuilt facebook.com](https://engineering.fb.com/2020/05/08/web/facebook-redesign/)
starting around 2019 and did not use React Router. Their custom router is
tightly coupled to their data-fetching layer through what they call
"EntryPoints" — each route definition statically declares what code and data it
needs, so the router can start fetching both on hover, before you've even
clicked. React Router had no concept of this until
[v6.4 introduced loaders](https://remix.run/blog/react-router-v6.4) in September
2022 — prior to that, routes discovered their data requirements during render,
which meant waterfalls.

Atlassian
[ran into the same wall](https://www.atlassian.com/blog/atlassian-engineering/react-resource-router-deep-dive)
from a different direction. They used React Router in Jira and needed SSR. React
Router's "dynamic routing" model — where routing happens during render rather
than before it — meant they had to walk the entire component tree before
rendering to discover data requirements, adding 20-50ms per server response.
When React Hooks landed, these workarounds broke entirely. They rewrote their
router, open-sourced it as
[`react-resource-router`](https://github.com/atlassian-labs/react-resource-router),
and reported a 250-300ms improvement in time to interactive. Dan Abramov had
warned against exactly this pattern: "trying to 'resolve' a react tree from a
library is a bad idea in the longer term. i mean projects that attempt to walk
the tree, instantiate components, call render methods, etc." React Router's
architecture had pushed them into doing exactly that.

Netflix took the most radical path in 2017: they
[removed client-side React](https://jakearchibald.com/2017/netflix-and-react/)
from their public landing page entirely. The landing page is a login form. It
doesn't need a router or a SPA. They kept server-rendered React for the initial
HTML, deferred the client-side JS, and got a 50% improvement in
time-to-interactive.

The catch-all rewrite pattern — the one most tutorials implicitly assume — is
not how any of these companies operate. It's a workaround that got normalized
without ever being properly named.

So it's worth asking: why not just make a real multi-page app?

The usual answer is "no page flash" and "preserved state across navigations."
I'm skeptical of both. The flash _is_ the feedback. It's the browser telling the
user something happened. SPA route changes often feel like a "soft click" — you
click a link, nothing visually resets, and you're not quite sure if it worked.
Real navigation gives you the tab spinner, the ability to hit stop, correct back
button behavior, and scroll restoration that actually works. These are things
SPAs have to painstakingly rebuild, and most don't get them right.

As for preserved state — a well-designed app usually doesn't need it. If your
page needs data, fetch it. If your user needs to stay logged in, use a cookie.
The cases where you genuinely can't survive a page reload are narrow (a music
player that keeps playing, a live websocket feed). That's not most apps.

What tends to happen instead is that people get so bought into the idea of React
as a full application platform that they try to use React Router to simulate
multi-page navigation inside a single page. The result is a system that is
harder to reason about than either a real SPA or a real MPA.

Multi-page apps are liberating. Each URL is a real server resource. You get
browser navigation for free. Error pages can return actual HTTP 404s. Search
engines don't need special treatment. You can still use React on each page if
you want — nothing stops you.

If you've thought about all of this and you still need React Router, go for it.
There are real use cases. But it's worth being honest that most of the time,
people reach for it out of habit rather than necessity.
