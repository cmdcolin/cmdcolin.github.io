<p>It is common in typescript to write <code>Record&#x3C;string,T></code> for whatever type <code>T</code> you
store in an object.</p>
<p>But, are you sure you're actually going to get an <code>T</code> back? What if some random
user provided string is used to index the Record, and it produces undefined? You
might think, as a typescripter "do I really have to add undefined to the
signature, shouldn't Typescript check this for me"? Well..</p>
<h2 id="nouncheckedindexedaccess"><a aria-hidden="true" tabindex="-1" href="#nouncheckedindexedaccess"><a href="#nouncheckedindexedaccess" style="margin-right: 10px">#</a></a><code>noUncheckedIndexedAccess</code></h2>
<p>There is actually a built-in Typescript setting called
<a href="https://www.typescriptlang.org/tsconfig#noUncheckedIndexedAccess"><code>noUncheckedIndexedAccess</code></a>
that will basically change your usage of a <code>Record&#x3C;string,T></code> into a
<code>Record&#x3C;string,T|undefined></code>. It actually makes the return type of accessing
both objects and arrays into an <code>T|undefined</code>.</p>
<p>The catch?</p>
<ul>
<li>It is default false</li>
<li>It is probably too "annoying" (or produces too many "false positives") for
most projects to warrant it being turned on</li>
<li>Hence, very few people use this setting</li>
</ul>
<p>Therefore: you really do have to ask yourself. Do you feel lucky punk? Sorry
wrong line. I mean, do you want to use a <code>Record&#x3C;string,T></code> or a
<code>Record&#x3C;string,T|undefined></code>?</p>
<p>Enjoy! <a href="https://www.typescriptlang.org/tsconfig#noUncheckedIndexedAccess">https://www.typescriptlang.org/tsconfig#noUncheckedIndexedAccess</a></p>
<p>Discussion on why this is not on by default:
<a href="https://github.com/microsoft/TypeScript/issues/49169">https://github.com/microsoft/TypeScript/issues/49169</a></p>
<h2 id="footnote-1-should-nouncheckedindexedaccess-be-split-into-two-settings"><a aria-hidden="true" tabindex="-1" href="#footnote-1-should-nouncheckedindexedaccess-be-split-into-two-settings"><a href="#footnote-1-should-nouncheckedindexedaccess-be-split-into-two-settings" style="margin-right: 10px">#</a></a>Footnote 1: Should <code>noUncheckedIndexedAccess</code> be split into two settings?</h2>
<p>I feel like using 'untrusted input' or 'runtime values' to access a Record is
more common than using 'untrusted input' to access an array index, so it would
be sort of nice if <code>noUncheckedIndexedAccess</code> was split into two separate
settings...one for Records, one for Arrays. A common complaint is that the
'array access' part of this setting is a little too strict, and requires
non-null assertions</p>
<h2 id="footnote-2-consider-using-a-map-instead"><a aria-hidden="true" tabindex="-1" href="#footnote-2-consider-using-a-map-instead"><a href="#footnote-2-consider-using-a-map-instead" style="margin-right: 10px">#</a></a>Footnote 2: Consider using a <code>Map</code> instead</h2>
<p>The <code>Map::get</code> method returns <code>T|undefined</code>, so it forces you to consider the
undefined case always. I find Maps slightly more cumbersome to use than plain
old objects, but has this benefit amongst others!</p>
<h2 id="foonote-3-small-downside"><a aria-hidden="true" tabindex="-1" href="#foonote-3-small-downside"><a href="#foonote-3-small-downside" style="margin-right: 10px">#</a></a>Foonote 3: Small downside</h2>
<p>Maybe you are certain there aren't actually undefined values stored in your
object, and you just used <code>Record&#x3C;string, T|undefined></code> to be able to say
<code>obj['randomthing']</code> returns undefined. Well, now calling e.g.
<code>Object.values(obj)</code> will return <code>(T|undefined)[]</code> (similar for
<code>Object.entries</code>) which is slightly annoying as a user, why may want it to just
be <code>T[]</code> but they can non-null assert or filter out potential undefined's as a
workaround.</p>
<h2 id="footnote-4-small-upside"><a aria-hidden="true" tabindex="-1" href="#footnote-4-small-upside"><a href="#footnote-4-small-upside" style="margin-right: 10px">#</a></a>Footnote 4: Small upside</h2>
<p>If you read this article and said, ya, I'm gonna use <code>noUncheckedIndexedAccess</code>
in the tsconfig.json of my library, and you feel all safe and cozy using
<code>Record&#x3C;string, T></code>, well your users may still have trouble: if the consumers of
your library don't use the <code>noUncheckedIndexedAccess</code> tsconfig.json setting,
then they will not get the benefit of that undefined condition unless you
